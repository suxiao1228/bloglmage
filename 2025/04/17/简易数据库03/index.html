引用计数缓存框架

实现缓存框架的策略

实现缓存框架时要考虑的一个点是缓存策略，这里采用的策略是使用引用计数。当某个资源被使用到时，加入到缓存，如果有其他的模块也在访问这个缓存，那么计数就加1，只有当所有的模块都释放了该资源时，将其从缓存中释放。

原文

WHY NOT LRU?

由于分页管理和数据项（DataItem）管理都涉及缓存，这里设计一个更通用的缓存框架。
看到这里，估计你们也开始犯嘀咕了，为啥使用引用计数策略，而不使用 “极为先进的” LRU 策略呢？
这里首先从缓存的接口设计说起，如果使用 LRU 缓存，那么只需要设计一个 get(key) 接口即可，释放缓存可以在缓存满了之后自动完成。设想这样一个场景：某个时刻缓存满了，缓存驱逐了一个资源，这时上层模块想要将某个资源强制刷回数据源，这个资源恰好是刚刚被驱逐的资源。那么上层模块就发现，这个数据在缓存里消失了，这时候就陷入了一种尴尬的境地：是否有必要做回源操作？

  1. 不回源。由于没法确定缓存被驱逐的时间，更没法确定被驱逐之后数据项是否被修改，这样是极其不安全的
  2. 回源。如果数据项被驱逐时的数据和现在又是相同的，那就是一次无效回源
  3. 放回缓存里，等下次被驱逐时回源。看起来解决了问题，但是此时缓存已经满了，这意味着你还需要驱逐一个资源才能放进去。这有可能会导致缓存抖动问题

当然我们可以记录下资源的最后修改时间，并且让缓存记录下资源被驱逐的时间。但是……
如无必要，无增实体。 —— 奥卡姆剃刀
问题的根源还是，LRU 策略中，资源驱逐不可控，上层模块无法感知。而引用计数策略正好解决了这个问题，只有上层模块主动释放引用，缓存在确保没有模块在使用这个资源了，才会去驱逐资源。
这就是引用计数法了。增加了一个方法 release(key)，用于在上册模块不使用某个资源时，释放对资源的引用。当引用归零时，缓存就会驱逐这个资源。
同样，在缓存满了之后，引用计数法无法自动释放缓存，此时应该直接报错（和 JVM 似的，直接 OOM）

如何实现缓存计数

AbstractCache

在common包中定义了一个AbstractCache抽象类，以及两个抽象方法，留给实现类去实现具体的操作

引用计数

除了普通的缓存功能之外，还需要维护另外一个计数。除此之外，为了应付多线程的场景，还需要记录哪些资源从数据源获取中。

  1. **private HashMap<Long, T> cache;**：这是一个 HashMap 对象，用于存储实际缓存的数据。键是资源的唯一标识符（通常是资源的ID或哈希值），值是缓存的资源对象（类型为 T）。在这个缓存框架中，cache 承担了普通缓存功能，即存储实际的资源数据。
  2. **private HashMap<Long, Integer> references;**：这是另一个 HashMap 对象，用于记录每个资源的引用个数。键是资源的唯一标识符，值是一个整数，表示该资源当前的引用计数。引用计数表示有多少个模块或线程正在使用特定的资源。通过跟踪引用计数，可以确定何时可以安全地释放资源。
  3. **private HashMap<Long, Boolean> getting;**：这是第三个 HashMap 对象，用于记录哪些资源当前正在从数据源获取中。键是资源的唯一标识符，值是一个布尔值，表示该资源是否正在被获取中。在多线程环境下，当某个线程尝试从数据源获取资源时，需要标记该资源正在被获取，以避免其他线程重复获取相同的资源。这个 getting 映射用于处理多线程场景下的并发访问问题。

下面这段话的理解（回源）

从缓存中获取资源Get
Get的基本流程就是进入死循环，不断尝试从缓存中获取。首先要判断是否有其他线程正在从数据源中获取这个资源，如果有的话就待会再看，否则：

如果资源已经在缓存中，那么就直接获取并返回，同时给资源的引用数+1
如果资源不在缓存中，那么就在getting中标记，然后该线程从数据源中获取资源
释放缓存Release
释放缓存时，将对应的引用计数-1，如果已经减到0了就回源，然后删除缓存中相关的各种数据

并不是很理解上面这些话，获取缓存为什么要+1,释放缓存为什么要-1,为什么为0的时候就回源，回源是什么意思？？？

解答：

想象一个简单的场景：你有一个图书馆（缓存），里面有一些书（缓存的资源 T）。

  1. 获取缓存 Get (get(key))
    • 流程: 你想借一本书（获取 key 对应的资源）。
    • 检查是否别人正在取: 你先看看有没有图书管理员正在帮你从仓库（数据源）拿这本书 (getting.containsKey(key))。如果有，你得等等。
    • 书在馆内 (cache.containsKey(key)): 如果书就在图书馆的书架上：
      • **图书管理员拿出书给你 (T obj = cache.get(key))**。
      • 关键：登记借阅 (references.put(key, references.get(key) + 1)): 图书馆需要记录这本书现在有多少人借走了。你借走了一本，所以这本书的“借阅人数”要加 1。
      • 为什么+1？ 为了知道这本书当前有多少个活跃的使用者。只要这个计数大于 0，就意味着至少还有一个人（或一个程序部分）正在使用这本书，图书馆就不能把它处理掉（比如卖掉或放回仓库不再提供）。
    • 书不在馆内 (Cache Miss): 如果书不在书架上：
      • 告诉管理员你要借这本书 (getting.put(key, true)): 防止其他人也同时让管理员去找同一本书。
      • 管理员去仓库拿书 (obj = getForCache(key)): 这是实际从数据源（比如数据库文件）加载数据的过程。
      • 拿回来后:
        • **放到书架上 (cache.put(key, obj))**。
        • 登记你是第一个借阅人 (references.put(key, 1)): 因为你是第一个拿到这本书的人，所以初始借阅计数是 1。
  2. 释放缓存 Release (release(key))
    • 流程: 你读完了书，要把书还给图书馆（释放对 key 对应资源的引用）。
    • 关键：注销借阅 (int ref = references.get(key)-1): 图书馆需要更新记录，表明借走这本书的人数减少了 1。
    • 为什么-1？ 因为你这个使用者不再需要这本书了，你归还了你的“引用”。
    • 检查是否还有人借阅 (if(ref == 0)):
      • 是 (ref == 0): 如果减去你之后，这本书的借阅人数变成 0 了，意味着现在没有人持有或使用这本书了。
      • 关键：“回源” (releaseForCache(obj)): 这时候图书馆就可以对这本书做最终处理了。“回源”在这里的意思是触发与数据源相关的清理或写回操作。具体含义取决于 releaseForCache 的实现：
        • 写回修改 (Write-Back): 如果你在书上做了笔记（修改了缓存数据），图书馆需要把这些笔记同步回原始记录或永久存档（将修改写回数据库文件）。
        • 释放资源: 如果这本书占用了特殊资源（比如一个需要手动关闭的文件句柄），现在可以安全地释放了。
        • 从图书馆移除 (references.remove(key); cache.remove(key); count –;): 处理完“回源”操作后，这本书就可以从图书馆的书架上移除，不再占用缓存空间。
      • 否 (ref > 0): 如果减去你之后，借阅人数还不为 0，说明还有其他人正在使用这本书。图书馆不能现在就处理或移除这本书。它只需要更新一下借阅人数 (references.put(key, ref)) 即可。

总结一下关键点：

  • +1 on Get: 表示有一个新的使用者开始依赖这个缓存项。计数器追踪活跃的使用者数量。
  • -1 on Release: 表示一个使用者结束了对这个缓存项的依赖。
  • Why remove only at 0? 因为只有当没有任何活跃使用者（计数为 0）时，移除缓存项才是安全的，不会影响到还在使用它的人。
  • “回源” (Hui Yuan - Return to Source): 指的是在缓存项可以被安全移除时（引用计数为 0），执行那些需要与原始数据源交互或进行最终清理的操作（通过调用 releaseForCache 方法实现），比如将缓存中的修改写回到数据库文件，或者释放底层资源。

所以这里用的是releaseForCache()；直接写回去然后清除这个缓存，那还在引用这个buffer的其他对象怎么办？那不就是空指针问题了？？？（记得解决）

直接在 close() 中为所有缓存项调用 releaseForCache() 并移除，确实会导致持有这些缓存项引用的其他线程出现严重问题，最典型的就是空指针异常（NullPointerException）或使用已被非法处理的资源。

我们来拆解一下这个问题：

  1. close() 的意图 vs. 现状：
    • 意图（合理推测）： 在关闭缓存时，确保所有缓存中的数据（特别是可能被修改过的“脏”数据）有机会被持久化或清理。releaseForCache() 被假定是执行这个操作的地方。
    • 现状： close() 方法简单粗暴地遍历 cache，对每个找到的对象 obj 调用 releaseForCache(obj)，然后无条件地从 cache 和 references 中移除对应的条目。它完全无视了 references.get(key) 的值。
  2. 引用计数的契约被打破：
    • get() 方法返回一个对象 obj 的引用，并通过增加 references.get(key) 来保证：只要调用者持有这个引用并且没有调用 release(key)，这个 obj 就应该是有效的、存在于缓存中的（至少逻辑上）。
    • release(key) 方法遵循这个契约，只有当引用计数降为 0 时，才认为没有任何线程在使用这个 obj 了，此时才安全地调用 releaseForCache(obj) 并将其从缓存中彻底移除。
    • close() 方法的操作 直接违反 了这个契约。它在引用计数可能大于 0 的情况下，就调用了 releaseForCache 并移除了缓存项。
  3. 后果：空指针和其他并发问题
    • 场景：
      • 线程 A 调用 get(buffer1)，获得 obj1，此时 references.get(buffer1) 计数增加（例如变为 50）。
      • 线程 B（可能是关闭钩子线程）调用 close()。
      • close() 遍历到 buffer1，获取 obj1。
      • close() 调用 releaseForCache(obj1)。这个方法可能会：
        • 将底层资源（如文件句柄、网络连接）关闭。
        • 将内存标记为无效。
        • 在数据库场景下，将“脏页”写回磁盘。
      • close() 从 cache 和 references 中移除 buffer1。
      • 此时，线程 A 仍然持有 obj1 的引用，并且它的引用计数（虽然已被 close 强制移除）逻辑上还未归零。
      • 线程 A 稍后尝试使用 obj1（例如读取数据、调用方法）。
    • 可能发生的问题：
      • NullPointerException: 如果 releaseForCache 将 obj1 内部的某个关键字段设为 null，或者线程 A 后续需要再次通过 cache.get(buffer1)（虽然设计上不应该，但可能发生）获取对象，会失败。更常见的是，如果线程 A 后来调用 release(buffer1)，会因为 references.get(buffer1) 返回 null（因为它被移除了）而抛出 NPE。
      • 使用已释放资源： 如果 releaseForCache 关闭了 obj1 使用的底层资源，线程 A 对 obj1 的操作可能会失败或导致未定义行为（例如，向已关闭的文件句柄写入）。
      • 数据不一致： 如果 releaseForCache 只是写回数据，但 obj1 本身结构还“可用”，线程 A 可能读取到的是一个已经被移除或即将被垃圾回收的对象的状态，这可能不是它期望的状态。

如何解决或回答这个问题？

你需要向面试官展示你理解了这个并发冲突，并能提出改进方案：

回答思路：

  1. 确认问题： “您是对的，我完全理解了。当前的 close() 实现确实存在严重问题。它直接调用 releaseForCache() 并移除缓存项，忽略了引用计数，这会打破引用计数机制建立的线程安全契约。”
  2. 解释后果： “正如我担心的，如果 close() 被调用时，还有其他线程持有缓存对象的引用（引用计数 > 0），那么这些线程后续访问该对象或调用 release() 时，极有可能遇到 NullPointerException、使用到已被错误处理的资源，或者数据不一致的问题。因为 close() 强制移除了对象，而其他线程对此并不知情。”
  3. 讨论 releaseForCache 的角色： “如果 releaseForCache 的目的是持久化脏数据（比如写回磁盘），那么在关闭时触发这个动作本身是合理的。问题在于触发的时机和后续的强制移除操作。”
  4. 提出解决方案：
    • 方案一：极其有限的场景（几乎不可行）： “这种 close() 只能在确保没有任何其他线程持有缓存引用的情况下安全调用，例如整个应用停止前的最后一步。但这大大限制了它的可用性。”
    • 方案二：优雅关闭 (Graceful Shutdown - 等待引用归零)： “更健壮的方法是实现优雅关闭。close() 应该：
        1. 设置一个“正在关闭”的标志，阻止新的 get() 请求成功。
        1. 遍历缓存中的所有条目。
        1. 等待 每个条目的引用计数自然下降到 0 （通过其他线程调用 release()）。这可能需要轮询、使用 Condition 等待，或者结合超时机制。
        1. 当一个条目的引用计数变为 0 时，才安全地调用 releaseForCache() 并将其从缓存中移除。
        1. close() 方法直到所有缓存项都被安全移除后才返回。”
      • 缺点： 如果存在引用泄漏（有线程 get 了但从未 release），close() 可能会永远等待下去。需要超时或强制策略。
    • 方案三：分离 Flush 和 Close (如果 releaseForCache 主要是 Flush)：
      • “可以将‘刷脏’和‘关闭’分离。
        1. 提供一个 flushAll() 方法，它遍历缓存，对所有（可能是脏的）对象调用 flush()（这可能就是 releaseForCache 的部分或全部功能），但不移除缓存项，也不检查引用计数。这确保数据被写回。
        1. close() 方法随后执行上述‘优雅关闭’逻辑（等待引用归零），但在引用归零时调用的清理方法（可能是 releaseForCache 或另一个方法）可能只需要做资源释放，因为数据已经在 flushAll 中处理过了。”
    • 方案四：修改 releaseForCache 语义 或 引入新方法：
      • “或者，明确 releaseForCache(T obj) 的职责。如果它只是写回数据，那 close 可以先调用它（或 flush），然后安全等待。如果它包含资源释放，那它只能在引用为0时被调用。可能需要一个 forceFlush(T obj) 方法给 close 调用，以及一个 finalCleanup(T obj) 给 release 在引用为0时调用。”
  5. 总结： “总之，当前的 close() 实现对于一个并发缓存来说是危险的。需要采用更安全的策略，如优雅关闭（等待引用计数归零）或分离刷脏与关闭逻辑，来确保在关闭缓存时不会破坏其他线程持有的有效引用，避免空指针等并发问题。”
